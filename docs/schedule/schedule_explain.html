GraphScheduler::schedule()

推理引擎调度入口 — 将计算图分配到多个设备并创建内存池

1
估算每层内存开销
estimate_layer_costs(graph)
遍历计算图中所有 Transformer 层，统计每层三类开销：
- 权重 (weight)：模型参数，只读
- 激活 (activation)：中间计算结果，层间复用
- KV Cache：注意力 Key/Value 缓存，由 max_seq_len 和 n_kv_heads 推算
激活乘以 activation_pool_factor 系数做安全预留。
厨师清点食材：这道菜需要多少肉、多少蔬菜、多少调料。
2
贪心分配连续层到设备
assign_layers(costs, devices)
设备按可用内存从大到小排序。从第一层开始，依次将连续层塞入当前设备：
每层占用 = weight + kv_cache + max(activation)
超出设备内存时切到下一个设备。CPU 作为最后兜底，若 CPU 也放不下则抛出异常。
返回值：每段一个 LayerAssignment{start_layer, end_layer, device, dev_id}
把衣服按顺序往衣柜里塞，塞满了换下一个衣柜，最后塞进储物间。
3
写入 Tensor 设备信息
apply_assignment(graph, assignments)
将分配结果落实到每个 Tensor::device 字段。
同时调用 mmanager_->register_layer_device(layer_id, dev_id)
在 MemoryManager 中注册 layer → device ID 的映射关系。
给每份食材贴上标签：这份放冰箱，那份放冰柜。
4
全局节点分配到 CPU
assign_global_nodes(graph, cpu)
layer_id < 0 的节点（如 rope_sin / rope_cos 旋转位置编码）不属于任何层，统一放在 CPU 上。
公共调料放在厨房台面上，不需要放冰箱。
5
插入跨设备拷贝节点
insert_copy_edges(graph)
当某层输入张量来自另一个设备时，在图中自动插入 memcpy 节点进行数据搬运。
去重机制：同一对 (src, dst_device) 只插入一次，多处引用复用同一拷贝节点。
拷贝节点的内存开销在第 6 步 create_memory_pools 中从 KV Cache 池扣减。
冰箱里的肉要做菜，先搬到灶台上（memcpy），再下锅。
6
创建内存池
create_memory_pools(graph, devices)
在每个设备上调用 mmanager_->get_or_create(dev, dev_id, weight, activation, kv_cache)，
创建三类内存池：MemoryPool（weight / activation / kv_cache）。
内存池内部预分配一整块连续内存，后续通过偏移量分配，零碎片。
剩余空间全部给 KV Cache，最大化可用上下文长度。
搬进新家前先定制好橱柜，尺寸精确到厘米，不留浪费空间。
7
初始化 KV Cache 分页管理器
initialize_kv_cache()
为每个设备创建 PagedAttentionManager，管理 KV Cache 的分页分配。
分页大小 = PAGE_BLOCK_SIZE（通常 16 tokens）
最大页数 = ceil(max_seq_len / PAGE_BLOCK_SIZE)
每层的注意力头独立管理，支持动态分配和回收。
图书馆给每个书架编号，借书还书按页码管理，高效又不浪费。

核心设计思想

  • 层间内存复用：激活池取单层最大值而非累加，推理是逐层执行的，前一层用完立即回收
  • 连续层分段：同一设备分配连续层，减少跨设备通信次数
  • 零碎片内存池：预分配大块连续内存，通过偏移量分配，避免 malloc/free 碎片化
  • 内存预算公式：need = Σweight + Σkv_cache + max(activation)，超出时自动切设备
  • CPU 兜底：GPU 内存不够时自动退到 CPU，保证模型一定能跑起来
