GraphScheduler::schedule()

推理引擎调度入口 — 将计算图分配到多个设备并创建内存池

1
估算每层内存开销
estimate_layer_costs(graph)
遍历计算图中所有 Transformer 层，统计每层三类开销：
- 权重 (weight)：模型参数，只读
- 激活 (activation)：中间计算结果，层间复用
- KV Cache：注意力 Key/Value 缓存，由 max_seq_len、PAGE_BLOCK_SIZE、n_kv_heads、head_dim 和 dtype 推算
激活乘以 activation_pool_factor 系数做安全预留；KV Cache 乘以 kv_cache_pool_factor。
注意：这里只是分层估算，第 6 步会在插入 copy 节点后重新按设备统计实际池大小。
厨师清点食材：这道菜需要多少肉、多少蔬菜、多少调料。
例子：手算第 0 层的 LayerCost
假设第 0 层是一个 BF16 Transformer 层，配置为 max_seq_len=8192、PAGE_BLOCK_SIZE=16、n_kv_heads=8、head_dim=128、activation_pool_factor=1.2、kv_cache_pool_factor=1.0。
estimate_layer_costs(graph) 会从 graph->get_layer_groups() 拿到这一层的计算张量，从 graph->get_all_tensors() 里单独累加这一层的权重。
统计对象 示例张量 计算方式 结果
activation_bytes x_norm、q_flat、k_flat、v_flat、attn_4d 跳过 view 张量后，把每个张量的 bytes_at(max_seq_len) 相加，再乘以 activation_pool_factor 576.0MB
weight_bytes attn_q.weight、attn_k.weight、attn_v.weight、ffn_*.weight 遍历所有 TENSOR_TYPE_WEIGHT，按 layer_id 累加 bytes() 30.0MB
kv_cache_bytes OP_TYPE_PAGED_ATTN 2 * ceil(8192 / 16) * 16 * 8 * 128 * 2 bytes 32MB
最终第 0 层得到：
与日志中的 L 0: weight=30.0 MB activation=576.0 MB kv_cache=32.0 MB total=638.0 MB 对应：
LayerCost{layer_id=0, weight_bytes=30.0MB, activation_bytes=576.0MB, kv_cache_bytes=32.0MB}
后续 assign_layers() 会用 total = weight + activation + kv_cache = 638.0MB 参与设备分段决策。
2
贪心分配连续层到设备
assign_layers(costs, devices)
非 CPU 设备按可用内存从大到小排序，CPU 只作为最后候选设备。
每个设备的可用预算不是全部空闲内存，而是：available_memory * (1 - memory_headroom)。
从第一层开始，依次将连续层塞入当前设备，段内估算公式为：
need = Σweight + Σkv_cache + max(activation)
其中 activation 取段内最大单层值，因为层间激活可复用。超出预算时切到下一个设备。
CPU 只有在设备列表中存在时才可兜底；没有可用设备或单层放不下当前设备都会抛出异常。
返回值：每段一个 LayerAssignment{start_layer, end_layer, device, dev_id, kv_cache_bytes, weight_bytes, activation_bytes}
把衣服按顺序往衣柜里塞，塞满了换下一个衣柜，最后塞进储物间。
例子：assign_layers 如何形成设备分段
假设 estimate_layer_costs() 得到 4 层，每层都和第 0 层类似：weight=30MB、activation=576MB、kv_cache=32MB。
设备列表里有一张 CUDA 卡和 CPU，memory_headroom=0.05。CUDA 可用内存为 820MB，实际调度预算为 820 * 0.95 = 779MB。
尝试加入层 段内 weight 段内 kv_cache 段内 max activation need 结果
L0 30MB 32MB 576MB 638MB 放入 CUDA:0
L1 60MB 64MB 576MB 700MB 继续放入 CUDA:0
L2 90MB 96MB 576MB 762MB 继续放入 CUDA:0
L3 120MB 128MB 576MB 824MB 超过 779MB，停止当前段
CUDA:0 最终只接收 L0 ~ L2，然后 assign_layers() 切到下一个候选设备 CPU，从 L3 重新开始累加。返回结果类似：
LayerAssignment{start_layer=0, end_layer=2, device=CUDA, dev_id=0, kv_cache_bytes=96MB, weight_bytes=90MB, activation_bytes=576MB}
LayerAssignment{start_layer=3, end_layer=3, device=CPU, dev_id=0, kv_cache_bytes=32MB, weight_bytes=30MB, activation_bytes=576MB}
关键点是 activation 不随层数累加，只保留当前段里的最大值；weight 和 KV Cache 会随层数累加。新设备会从失败的那一层重新开始尝试。
3
写入 Tensor 设备信息
apply_assignment(graph, assignments)
将分配结果落实到每个 Tensor::device 字段。
同时调用 mmanager_->register_layer_device(layer_id, dev_id)
在 MemoryManager 中注册 layer → device ID 的映射关系。
给每份食材贴上标签：这份放冰箱，那份放冰柜。
例子：apply_assignment 如何写回 Tensor::device
接着上面的分段结果：L0 ~ L2 放到 CUDA:0，L3 放到 CPU:0。
apply_assignment() 先展开每个连续段，得到一个临时表 layer_dev，再遍历图中所有 Tensor，把匹配到的层写入对应设备。
assignment 展开后的 layer_dev MemoryManager 注册
L0 ~ L2 → CUDA:0 layer_dev[0]=CUDA
layer_dev[1]=CUDA
layer_dev[2]=CUDA		 register_layer_device(0, 0)
register_layer_device(1, 0)
register_layer_device(2, 0)
L3 → CPU:0 layer_dev[3]=CPU register_layer_device(3, 0)
Tensor layer_id 写回前 写回后 原因
blk.0.attn_q.weight 0 CPU CUDA layer_dev[0] 命中 CUDA
blk.2.ffn_out 2 CPU CUDA layer_dev[2] 命中 CUDA
blk.3.attn_4d 3 CPU CPU layer_dev[3] 命中 CPU
rope_cos -1 CPU CPU layer_id < 0，这一步跳过，下一步统一处理全局节点
这一步只写 Tensor::device 和 MemoryManager 的 layer → dev_id 映射；它还不会分配真实内存，也不会插入 memcpy。
跨设备输入要等第 5 步 insert_copy_edges() 才会根据这些 Tensor::device 字段被发现。
4
全局节点分配到 CPU
assign_global_nodes(graph, cpu)
layer_id < 0 的节点（如 rope_sin / rope_cos 旋转位置编码）不属于任何 Transformer 层，统一写成 CPU。
这一步发生在跨设备拷贝节点插入之前，所以后续如果某个设备需要这些全局张量，会由第 5 步自动插入 CPU → 目标设备的 memcpy 节点。
公共调料放在厨房台面上，不需要放冰箱。
例子：assign_global_nodes 如何处理全局 Tensor
前面 apply_assignment() 只处理 layer_id >= 0 的层内 Tensor。现在轮到不属于任何层的全局 Tensor，例如 RoPE 查表、输入占位或输出相关辅助节点。
Tensor layer_id 写入设备 说明
rope_cos -1 CPU 全局 RoPE cache，不属于某个 Transformer layer
rope_sin -1 CPU 和 rope_cos 一样先固定在 CPU
input_ids -1 CPU 输入 Tensor 通常从 CPU 侧绑定
这一步的结果是：层内 Tensor 已经按 LayerAssignment 分到 CUDA/CPU，全局 Tensor 统一保持 CPU。
如果 CUDA 层后续读取 rope_cos，第 5 步会发现 rope_cos(CPU) → consumer(CUDA)，于是插入 CPU → CUDA 的 memcpy。
5
插入跨设备拷贝节点
insert_copy_edges(graph)
当某个 Tensor 的输入来自另一个设备时，在图中自动插入 memcpy 节点进行数据搬运，并把 consumer 的 src[] 改指向拷贝结果。
外部输出也会检查：如果最终输出不在 CPU，会额外插入回拷 CPU 的节点。
GPU → GPU 的拷贝会拆成两跳：GPU → CPU → GPU；CPU ↔ GPU 则是一跳。
去重机制：同一对 (src, dst_device) 只插入一次，多处引用复用同一拷贝节点。插入完成后会 rebuild_order()，保证拓扑顺序包含新节点。
拷贝节点的内存开销不会在第 1 步估算，实际会在第 6 步重新统计，并可能压缩 KV Cache 池。
冰箱里的肉要做菜，先搬到灶台上（memcpy），再下锅。
例子：insert_copy_edges 如何补跨设备边
现在图里已有设备标记：L0 ~ L2 在 CUDA，L3 在 CPU，全局 RoPE 节点也在 CPU。
insert_copy_edges() 遍历每个 Tensor 的 src[]，只要发现 src->device != consumer->device，就准备插入 memcpy。
发现的边 原因 插入节点 改写后引用
rope_cos(CPU) → L0.q_rope(CUDA) 全局 RoPE 在 CPU，L0 在 CUDA memcpy_rope_cos_to_cuda L0.q_rope.src = memcpy_rope_cos_to_cuda
rope_sin(CPU) → L0.q_rope(CUDA) 同上 memcpy_rope_sin_to_cuda L0.q_rope.src = memcpy_rope_sin_to_cuda
L2.layer_output(CUDA) → L3.x_norm(CPU) CUDA 段输出被 CPU 段消费 memcpy_layer_output_to_cpu L3.x_norm.src = memcpy_layer_output_to_cpu
如果多个 consumer 需要同一个 (src, dst_device)，cache 会复用同一个 memcpy 节点。
所有新节点插入后调用 graph->rebuild_order()，确保执行顺序从 “生产者 → memcpy → consumer”。
6
创建内存池
create_memory_pools(graph, devices)
这一步基于已经插入 memcpy 节点后的图重新按设备统计内存，而不是直接复用第 1 步的 per-layer 估算。
每个设备统计四类开销：weight_bytes、activation_bytes、kv_cache_bytes、copy_bytes。
权重池累加该设备上的所有权重；激活池按层统计并取该设备上的最大单层激活，再乘以 activation_pool_factor，且最小为 64MB。
KV Cache 池累加该设备上所有 paged attention 层需要的 cache。
如果 copy 节点占用了预算，会优先保留 weight + activation + copy，然后把 KV Cache 池压缩到剩余预算内；若固定开销已经超过预算，则抛出异常。
最后调用 mmanager_->get_or_create(dev, dev_id, weight, activation, kv_cache) 创建 weight / activation / kv_cache 三类池。
搬进新家前先定制好橱柜，尺寸精确到厘米，不留浪费空间。
例子：create_memory_pools 如何二次统计
第 6 步已经能看到 memcpy 节点，所以它会重新按设备统计真实池大小。继续使用前面的分段：CUDA 有 L0~L2，CPU 有 L3，并假设 CUDA 上新增 copy tensor 总计 64MB。
设备 weight activation copy kv_cache 最终创建
CUDA:0 90MB max(576MB, 64MB下限)=576MB 64MB 96MB get_or_create(CUDA, 0, 90MB, 576MB, 96MB)
CPU:0 30MB 576MB 32MB 32MB get_or_create(CPU, 0, 30MB, 576MB, 32MB)
如果某设备预算不足，先保留 weight + activation + copy，再把 KV cache 压缩到剩余空间。
例如预算 760MB，CUDA 固定开销 90 + 576 + 64 = 730MB，KV cache 最多只能保留 30MB，于是 96MB → 30MB。
7
初始化 KV Cache 分页管理器
initialize_kv_cache()
按 assignments_ 遍历每个设备段；只有该设备存在 kv_cache 池时才创建 PagedAttentionManager。
分页大小 = PAGE_BLOCK_SIZE（通常 16 tokens）
最大页数 = ceil(max_seq_len / PAGE_BLOCK_SIZE)
对段内每个 OP_TYPE_PAGED_ATTN Tensor，读取 n_kv_heads、head_dim 和 dtype，调用 init_layer() 初始化层元信息，再调用 reserve_layer() 预留页表容量。
这里做的是分页管理器初始化，为后续运行时动态分配/回收 KV block 做准备。
图书馆给每个书架编号，借书还书按页码管理，高效又不浪费。
例子：initialize_kv_cache 如何建立页表
假设 max_seq_len=8192、PAGE_BLOCK_SIZE=16，则每层最多需要 ceil(8192 / 16)=512 个 block。
初始化时按 assignment 找到每个设备上的 paged attention 层，然后给对应设备的 PagedAttentionManager 登记层信息。
assignment paged attention 层 init_layer reserve_layer
CUDA:0 L0 ~ L2 L0、L1、L2 init_layer(layer, 8, 128, BF16) reserve_layer(layer, 512)
CPU:0 L3 L3 init_layer(3, 8, 128, BF16) reserve_layer(3, 512)
这里 reserve 的是页表容量，不是把所有 token 的 KV 数据都立即写满。真正的 KV block 使用会在推理运行时按 token 进度分配。
8
打印调度摘要
print_summary(costs, devices)
schedule() 最后会打印调度结果，帮助确认前面几步的决策是否符合预期：
- 每层估算：weight / activation / kv_cache / total
- 设备分段：每个设备负责的层范围，例如 L0 ~ L15
- 总量统计：总权重、最大单层激活、总 KV Cache
这一步不改变图或内存池，只是输出可读摘要。
最后拿到一张装箱清单：每个箱子放了哪些东西、占了多少空间。
例子：print_summary 输出如何读
print_summary() 使用第 1 步的 per-layer cost 和第 2 步的 assignments 输出摘要。它不改变调度结果，只帮助你检查 “每层估算” 和 “设备分段” 是否符合预期。
示例输出：
L 0: weight=30.0 MB activation=576.0 MB kv_cache=32.0 MB total=638.0 MB
L 1: weight=30.0 MB activation=576.0 MB kv_cache=32.0 MB total=638.0 MB
CUDA : L:0 ~ L:2 (3 layers, exc186.0 MB, share96.0 MB)
CPU : L:3 ~ L:3 (1 layers, exc62.0 MB, share32.0 MB)
其中 exc 来自 weight_bytes + kv_cache_bytes，share 是该段的 kv_cache_bytes。

核心设计思想

  • 层间内存复用：激活池取设备内最大单层激活而非累加，推理是逐层执行的，前一层用完后激活池可复用
  • 连续层分段：同一设备分配连续层，减少跨设备通信次数
  • 零碎片内存池：预分配大块连续内存，通过偏移量分配，避免 malloc/free 碎片化
  • 预算留白：设备预算为 available_memory * (1 - memory_headroom)，避免把设备内存吃满
  • 内存预算公式：分层阶段使用 need = Σweight + Σkv_cache + max(activation)，超出时切设备
  • 图改写后二次统计：插入 memcpy 后重新统计真实池大小，copy 开销会挤占 KV Cache 预算
  • CPU 兜底有限：CPU 必须存在于设备列表中，且预算也必须容纳剩余层；否则调度会失败
